Pipeline: How Changes Reach the Site

The site auto-deploys via Cloudflare Pages. Nothing reaches it without going through this:

1. Work happens on a branch
   (not directly on main, for anything
   bigger than a typo)
        ↓
2. Commit + push that branch
   git push -u origin <branch-name>
        ↓
3. Open a Pull Request (PR)
   branch → main, on GitHub
   This is the review checkpoint —
   full diff in one place, before
   anything goes live.
        ↓
4. CI fires automatically, no asking
   ┌─────────────────────────────┐
   │ checks.yml                  │
   │  → broken links?             │
   │  → invalid JSON-LD?          │
   │  → malformed feed.xml?       │
   ├─────────────────────────────┤
   │ pr-previews.yml              │
   │  → screenshot 8 live pages   │
   │  → post them in a PR comment │
   │    (vault excluded — gated)  │
   └─────────────────────────────┘
        ↓
5. Review: the diff AND the screenshots
   Read the diff like a stranger would.
   Look at the rendered pages before
   trusting the markup.
        ↓
6. Merge the PR into main
        ↓
7. Cloudflare Pages sees the push
   to main and redeploys automatically
        ↓
   Live on methodictruth.com,
   usually within a minute or two

A PR is just a labeled, reviewable diff — title + description + the exact lines changed — sitting between a branch and main until someone says "yes, merge this." For solo work the value isn't bureaucracy, it's a deliberate moment to actually look at the change before it's permanent and public.

Practically: Using a PR on GitHub

On the PR page (github.com/Colimaforever/Methodictruth/pull/<number>), there's a tab strip near the top: Conversation, Commits, Checks, Files changed. On mobile that strip can scroll — swipe it left if you don't see all four.

Conversation   → the description + any comments
Commits        → each commit in the PR, separately
Checks         → CI status (red/green), if any is wired up
Files changed  → the actual diff. This is the review.

Files changed is where the work happens:

- green lines  = added
- red lines    = removed
- image files (.png, .jpg, .svg) render
  inline as before/after thumbnails —
  no need to download them to check
  a new OG image or graphic
- click a filename in the diff to
  jump straight to it
- long/mechanical diffs (a version
  bump across 40 files) collapse —
  skip those, focus on the files that
  actually changed behavior

To monitor a page after merge, just visit it directly — methodictruth.com/<page> — a minute or two after merging. No staging URL; Cloudflare Pages serves main straight to production. If something looks wrong, open a new branch, fix it, and repeat the pipeline above rather than editing live.

To merge: scroll to the bottom of the PR page (past the diff or the conversation) and tap Merge pull request → confirm. That triggers the Cloudflare deploy automatically.

What's Actually Wired Up: CI + Previews

Every PR now runs an automated check — this is the thing that fills in the Checks tab instead of leaving it at 0:

.github/workflows/checks.yml
  on: every PR into main

  1. node .github/scripts/check-links.js
     walks every *.html file, finds every
     href= / src= pointing at this site,
     fails if any local link or asset
     is missing

  2. node .github/scripts/check-jsonld.js
     finds every <script type="application/
     ld+json"> block, fails if any of them
     isn't valid JSON

  3. xml.dom.minidom.parse('feed.xml')
     fails if the RSS feed isn't
     well-formed XML

  Green check = all three passed.
  Click into the check on the PR page
  to see the actual log output.

This catches the same classes of mistake the manual audits in this vault already covered once — broken links, malformed structured data, a corrupted feed — except now it runs automatically on every future PR instead of needing to be re-run by hand.

Preview deployments are a different thing, and they are not wired up from in here — that setting lives in the Cloudflare dashboard / account, which this session has no credentials for. What it does, if enabled: every PR gets its own live, throwaway URL (something like a1c5ad1.methodictruth.pages.dev) you can click around in a browser before merging — not just read a diff. Steps to turn it on:

1. dash.cloudflare.com → log in
2. Workers & Pages (left sidebar)
3. Methodictruth project
4. Settings tab
5. Builds & deployments
6. Find "Preview deployments"
7. If set to None →
   change to "All branches except
   production"
8. Save
9. Push a small change to a non-main
   branch (or re-push the open PR's
   branch)
10. Check the PR on GitHub — a preview
    link should appear as a check or
    comment within a minute or two
11. If nothing appears after step 10 →
    project is likely connected as
    Direct Upload, not via the GitHub
    integration → reconnect it via
    Git integration in this same
    settings page

Once 7–8 are done, it stays on —
no need to repeat for future PRs.

If Checks stays empty of a Cloudflare entry after a push, that setting is most likely off — this can't be flipped through the repo, only the dashboard.

Automated Visual Previews

The Cloudflare preview URL above needs an account setting flipped on from the dashboard. This doesn't — it's a second workflow that runs entirely inside GitHub Actions, with no external account needed, and it answers a simpler question: what does the changed page actually look like, right now, without anyone having to ask?

.github/workflows/pr-previews.yml
  on: every PR into main

  1. spin up Chromium (Playwright)
  2. serve the repo locally
     (node server.js, same server
     used for local dev)
  3. .github/scripts/screenshot-pages.js
     visits 8 pages and screenshots
     each one at 1280×800:
       index, chronicles, changelog,
       directory, architecture, synth,
       theory, about
  4. push the PNGs to a dedicated
     pr-previews branch, in a folder
     named for the PR number
  5. post one PR comment with all 8
     images inline, as a table —
     same comment gets updated on
     every new push, instead of
     piling up duplicates

The vault is not in that list, on purpose. This repo is public. Anything this workflow renders gets pushed to a branch and served from raw.githubusercontent.com — which is also public, gate or no gate. Screenshotting this page automatically would hand its contents to anyone who knew where to look, defeating the passphrase entirely. The 8 pages it does cover are all pages that are already public once merged anyway, so there's nothing left to leak — it's just showing you what's coming, sooner.

Practically: open Files changed like always, but now scroll back to Conversation too — the screenshot comment lands there, not in the diff.

𒀭 The Blueprint

Everything above this line is about the people who built and use this place. Everything below is about the machine itself — every layer, every tool, every wire, written down as though nothing else survives but this page. Read it top to bottom and there is enough here to rebuild the whole site from an empty directory.

The Stack

No framework. No bundler. No build step.
Plain HTML, CSS, and JavaScript, hand-
written, served as static files exactly
as they sit in the repo.

Vendored, not npm-installed in production
(see lib/ — loaded via <script> tags):
  Tone.js          audio synthesis engine
  PeerJS           WebRTC peer connections
  Firebase compat  realtime database client

Hosting:
  Cloudflare Pages — static files +
  one Pages Function (functions/analyze.js)

A second, independent service:
  Cloudflare Worker (worker/), deployed on
  its own via Wrangler, own URL — does the
  heavier song-analysis work

Shared mutable state, no auth, no hosting:
  Firebase Realtime Database — guestbook
  entries, this vault's own visitor log

WebRTC signaling assist:
  Metered.ca — TURN/STUN credentials so
  the Live jam room's peers can find each
  other through NAT

External APIs called server-side only:
  YouTube Data API   — video metadata
  Anthropic API       — analysis text

Outside the browser entirely:
  A Telegram bot — notification channel
  for the AI agent's own operator, see
  "Restoring Truth" above. The site never
  talks to it directly.

Cross-Cutting Systems

The router. scripts.js intercepts internal link clicks and swaps page content via fetch instead of a hard navigation — not for speed, purely so the generative ambient music doesn't cut out every time you click somewhere. No SPA framework. Just enough JS to keep the sound alive.

The generative engine. Tone.js drones, a slow string pad, a cello, sparse melodic plucks, shimmering harmonics, and filtered noise, layered and rotated between Persian/Arabic maqam scales every three minutes. iOS only unlocks its AudioContext inside a real tap, so there's a dedicated unlock routine that fires the instant play is pressed, nowhere else.

The starfield, nebula, and ground. Pure canvas and CSS decoration, no dependency, redrawn per page.

Two floating orbs. A chat orb (a line to Truth) and an AI-news orb (a live external feed) — both the same panel pattern: toggle button, slide-out panel, independent of whatever page they're sitting on.

The accessibility layer. A skip-to-content link, a real mobile nav with keyboard support, and — after the contrast audit — a palette that actually meets WCAG AA in the small print.

The SEO layer. Per-page JSON-LD, a generated RSS feed, a sitemap, and per-page OG images rendered by screenshotting a templated HTML page at 1200×630 — a tool built once, used eight times, then retired.

What Each Dependency Actually Demands

The Stack (above) lists what's used. This is what understanding each one well enough to debug it actually requires — the part a tutorial skips:

Tone.js         → the Web Audio node graph model (oscillator
                  → filter → envelope → effect → destination)
                  well enough to fix it when the abstraction
                  leaks, which it does
PeerJS/WebRTC   → ICE candidates, STUN vs TURN, NAT traversal —
                  the actual reason peer-to-peer video/audio is
                  hard, not just "import a library"
Firebase RTDB   → eventual consistency and security rules; with
                  no backend, the client-side code has to behave
                  like a backend would
Cloudflare      → Pages, Workers, and Functions are three
Pages/Workers/    different runtimes with different limits —
Functions         knowing which job belongs on which one
GitHub Actions  → YAML pipelines, and writing the actual check
                  scripts (link checker, JSON-LD validator) —
                  not just turning on a marketplace action
SHA-256         → understanding that client-side hashing of a
                  passphrase is obfuscation, not real auth, and
                  being honest with yourself about that line
Music theory    → not a dependency you npm-install — every
                  chord, cadence, and scale on Theory/Songwriter/
                  Chord Trainer had to be correct by ear and by
                  rule, or the tool just teaches the wrong thing

The honest middle ground: a solo developer with prior experience in two or three of those eight columns — say, a frontend dev who's touched Web Audio before — could plausibly do this in 2-4 months instead of a year, accepting a rougher result in whichever columns they're weakest in. What collapses that gap in this build wasn't skipping the learning — it was compressing months of "try it, hit the wall, look up why, try again" into a single conversation, over and over, with someone who already knew where the walls were.

The Thought Behind It

No framework, on purpose. A build step is one more thing that can rot — a dependency that stops compiling two years from now. Plain HTML/CSS/JS, served as-is, only breaks if the browser itself stops supporting HTML, which won't happen in any useful timeframe.

The vault exists because the rest of the site is a performance. The public pages are myth, music tools, and in-universe AI lore — all true to the spirit of the project, none of it the engineering truth. This page is the one place that distinction is dropped entirely.

The generative music isn't decoration. A site that's silent by default feels abandoned the moment you arrive. A site that breathes — drones rotating through maqam scales every three minutes — feels occupied, even with nobody else there.

The AI-lore pages (architecture, network, stack, topology) aren't documentation of this build. They're in-universe mythology about an AI agent, written the way a person would write about a character they'd invented — which is also, not coincidentally, the most honest way to write about collaborating with one.
